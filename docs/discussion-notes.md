## Session 1 — Scoping (Jul 13)
- **Crux identified:** schema design under ambiguity. The brief deliberately refuses to specify what to extract; the structured representation of a repair note is the most consequential (and most interviewed) decision.
- **Second crux:** value is in *aggregation*, not per-note reading — analysts already read single notes; they miss batch-level patterns.
- **Planted signal in the data:** 5/30 notes are CR-V infotainment/display failures (RO-100001, -100006, -100010, -100016, -100026) across MY2022–24, low mileage, escalating to safety-relevant (backup camera). Dashboard must surface this cluster fast.
- **Warranty is a spectrum, not a label:** notes split three ways — explicitly covered, explicitly denied (4 distinct reasons: wear item / out of warranty / external cause / customer-caused), and silent/undecided (NFF, declined, escalated). A binary field forces hallucinated decisions → 4-state enum + denial_reason.
- **Rate limit as architecture forcing function:** OpenRouter free tier ≈ 20 req/min, 50 req/day. 30 notes ⇒ ~1 LLM call/note + small aggregate budget. Rules out multi-call agent loops per note (30×3=90 calls = dead on first run). Retries must be capped and budgeted.

## Session 1 — Scaling interpretation
- Two readings of "this does not scale": (1) throughput/velocity, (2) schema generalization.
- **Locked: throughput scaling.** Target = batch-processing velocity vs. manual analyst reading. Schema is prototype-scoped (this dataset, these models) but designed with **generalization seams** we can argue about hardening (canonical parts taxonomy, schema versioning, budgeted schema generation).
- Consequences: schema is the contract → design it right before writing the prompt; unparseable notes marked "needs review", never guessed; per-note LLM calls (batching trades blast radius for savings we don't need); S3 cache doubles as audit trail; dashboard is batch/aggregation-first, note-detail second.

## Session 1 — EDA module framing
- EDA module uses **zero LLM calls** (pandas/regex, optional spaCy/HF) — schema must be an outcome of interrogating the data, not LLM freestyling.
- Not strictly crucial for the assignment, but included because it demonstrates the schema-hardening story and costs little.
- **Framing locked: "schema validation", not "schema generation".** Generation on 30 notes over-claims; validation ("does the hand-derived schema cover the corpus?") is honest and defensible. Hardening arc: validation (prototype) → generation (production) — same module, growing capability.

## Talking point — budgeted LLM-based EDA (keep for project doc)
### Schema lifecycle: validation today, generation at scale
**Prototype (current):** The extraction schema is hand-derived from the 30-note sample. A validation module (regex lexicons + optional NLP models, no LLM calls) runs over the full dataset and verifies coverage: every component mention, warranty phrase, and severity indicator in the raw text maps to a schema category. Unmapped terms are reported, not silently dropped. This catches schema drift before extraction runs.

**Production (hardening path):** At 10K+ notes, hand-deriving a schema is infeasible and pure-NER validation misses semantics. The validation module grows into a schema *generation* module using **budgeted LLM-based EDA**:
- **Stratified sampling, not full-corpus reads.** Sample notes across model × model-year × dealer region (e.g., 200 of 10,000). The LLM never sees the full corpus — cost stays fixed as data grows.
- **Cheap pass first, LLM pass second.** Embeddings + clustering group similar notes; the LLM labels *cluster representatives* only (e.g., 30 calls for 30 clusters, not 10,000 calls). The label set becomes candidate schema categories.
- **Convergence as a stopping rule.** Sample in rounds; when a new round yields no new categories, stop. LLM budget is spent proportional to schema novelty, not corpus size.
- **Output is a versioned schema artifact.** Generated schema is diffed against the current version; new categories (e.g., a new EV battery failure mode) trigger review before extraction adopts them. Extraction always conforms to a locked schema version — generation and extraction never mix budgets.

## Session 1 — Validation findings (empirical)
- 100% subsystem coverage on the 30 notes with a 10-value enum; several notes legitimately touch multiple subsystems → extractor picks the *primary*.
- Warranty signals: 6 covered, 5 denied, 11 undetermined-with-signal, **8 completely silent** → empirical justification for the `undetermined` state.
- Keyword brittleness observed directly: "not warranty." false-matched covered pattern "warranty." until denied-spans were masked first → concrete evidence keyword rules can validate signal presence but cannot decide per-note fields; justifies the LLM extractor.
- All schema flags (repeat_visit, safety_related, customer_distress, vehicle_disabled, fleet_signal) trigger ≥3 times in corpus → no dead weight in schema.

## Session 1 — HF models vs regex in EDA (talking point)
- HF zero-shot could replace regex pattern matching and wins on paraphrase/unseen phrasing, but loses on domain shorthand (OOW/BO/NFF are OOD for general NLI models), has uncalibrated scores (thresholds unpickable on 30 unlabeled notes), and a fine-tuned classifier needs labels we don't have.
- Deeper point is role, not accuracy: notebook regexes are validation instruments (presence checks — deterministic + auditable wins); semantic per-note decisions belong to the LLM extractor (which also knows the shorthand). An HF classifier between them would own no job.
- Locked layering: regex = deterministic signal detection (validation) | LLM = semantic decision (extraction) | HF embeddings = similarity structure (schema generation mode, deferred stretch).

## Session 1 — Semantic-driven discovery status + local LLM (talking points)
- Honest status: validation is hybrid (regex+semantic, V1–V6) today; discovery is lexicon-based with the monotonic-extension rail built (Part 3b, compat-proven); the bridge — V6 low-coverage/regex-silent notes → cluster → candidate extensions into extend_schema() — is designed but deferred behind the core pipeline (stretch).
- Lightweight local Llama fits the discovery seat: weak model where mistakes are cheap (gated proposals), strong hosted model where they aren't (extraction). Local = no rate limit, no cost, no data egress for real notes. OpenAI-compatible endpoints make it a pure env-config swap.

### Session 2 — architecting phase (ratified decisions, pending formal SDD write-up as Decision E/F)
- **Aggregation = one grammar.** Every view is a deterministic pivot over the cached extraction set: assign each field a role — **Group / Filter / Measure** — plus optional `rank` + `top_k`. Serializable **query object** is the single contract; `/aggregate` is one compute-on-read endpoint, zero LLM. (Generalizes/retires the old §5 "menu of aggregations".)
- **Three surfaces over that one engine (all emit the same query object):** (1) **curated presets** — headline defect board + warranty/watch lenses, run on load = analyst cold-start; (2) **query panel** — pick actors/signals, set filters + top-k, ranked-table answer = power use (e.g. "top-10 models × top-3 subsystems, recurrent, on-warranty, MY2020–23"); (3) **visuals** — shape-appropriate chart (heatmap/stacked-bar/rate-bars/KPI) auto-selected by (#Group fields, is-ranked); table wins whenever there's a rank/top_k. A preset is just a saved query object (not a 4th thing).
- **Output-type rule:** 0 Group → KPI cards; 1–2 Group, no rank → heatmap/bars; any rank/top_k → ranked (nested) table. Honesty guardrails: always show `n`; prefer table when cells are thin; priority score is the ONE imposed opinion → keep formula/contributions transparent.
- **Priority score is a tunable Measure.** Default weights (sane, surfaces CR-V cluster) + named presets (safety-/recency-/volume-first); raw sliders = stretch. Weights live in the query object (shareable). Tuning changes the ranking lens, not the counts.
- **LLM is used ONLY at ingest** (raw note → schema record, 1 call/note, cached). All 3 surfaces are 100% deterministic, LLM-free. Deliberate non-use: no NL→query LLM (structured panel instead) — filed as "what's next," not a gap.
- **Scaling invariant:** the schema is the axis; records are points on it. Adding records under a FIXED schema is seamless (validate → cache → next compute-on-read includes them). Schema *extension* is monotonic (optional-only) but bumps `schema_version` → forces re-extraction (budget cost) + null new fields on old records. Demo: 1000-record set exercises the cheap side of that line.
- **Provider ladder (env-selected, `LLM_PROVIDER` seam):** Phase 1 **OpenRouter** `openrouter/free` (30 records, build+core demo, OpenAI-SDK adapter) → Phase 2 **Bedrock + Claude** (1000-record scaling run; own boto3/converse adapter; REAL AWS, not LocalStack → S3 stays emulated = hybrid story; own key lifts 50/day cap; ~single-$ cost; `meta.model` keeps the two sets traceable) → Phase 3 **Llama + vLLM** for RAG/NL ask-layer (self-hosted, OpenAI-compatible adapter; STRETCH, gated on core+tests). vLLM is the only path where we write serving code; Bedrock/Claude is a managed API (closed-weight, no vLLM). Name is **OpenRouter** (not "OpenQuery").
- **Decision G — async ingest / live-progress pipeline (RATIFIED):** decouple slow LLM ingest from fast deterministic serving. `POST /ingest` → 202 + job_id → SQS → worker(s) (rate-limited, idempotent via S3 cache-check) → per record: extract → validate → write S3 → DuckDB INSERT → job counter++; failures → needs_review + DLQ. Progress via polling `GET /jobs/{id}`; analytics refresh poll-while-active so the board re-ranks live (batch-health strip = live progress surface). Picks: SQS (in-process asyncio worker fallback behind same `extract()` interface) · S3 job manifest+counters (DynamoDB = scale) · polling (SSE = upgrade) · poll-while-active. It's the 1000-record live-demo centerpiece. Written into SDD §7.3 (+ §9 endpoints, §2 `ingest.py`/`worker.py`, env vars, build order, §12 AWS mapping).
- **Decision C clarified & extended in SDD §6:** field/enum descriptions are the semantic ground truth, authored once in the schema and consumed by BOTH the `json_schema` constraint (`extraction_schema.json` descriptions) AND the prompt (`schema_spec.yaml` `semantic_descriptions` for warranty/severity). Anchors (input calibration, other notes) ≠ `evidence_quote` (output grounding, a substring of THIS note) — orthogonal; the quote never cites an anchor.
